# DailyBill_2 项目方案文档

## 一、项目概述

### 1.1 项目背景
随着移动支付的普及，个人财务数据分散在支付宝、微信、银行卡等多个平台。手动整理和分类这些交易记录既耗时又容易出错。本项目旨在开发一款本地化的个人账单管理工具，帮助用户自动读取、分类和整理多平台的交易流水。

### 1.2 项目目标
- 实现支付宝、微信、银行交易流水的自动读取
- 基于交易描述智能匹配分类
- 输出标准化的分类账单 Excel 文件

### 1.3 目标用户
- 需要管理个人/家庭财务的用户
- 希望定期复盘消费习惯的用户
- 有记账需求但不想手动分类的用户

## 二、功能需求

### 2.1 核心功能

| 功能模块 | 功能描述 |
|---------|---------|
| 文件读取 | 支持读取支付宝、微信、银行导出的 Excel/CSV 交易流水文件 |
| 智能分类 | 基于交易描述/商户名称自动匹配到预定义类别 |
| 手动调整 | 支持用户手动修改自动分类结果 |
| 数据导出 | 输出整理好的 Excel 文件，按类别汇总 |

### 2.2 分类类别

项目支持将交易流水分类到以下 19 个类别：

| 序号 | 类别 | 说明 |
|-----|------|------|
| 1 | 旅行 | 机票、酒店、景点门票、旅游服务等 |
| 2 | 住房相关 | 房租、物业费、维修等 |
| 3 | 餐饮 | 外卖、餐厅消费、食材采购等 |
| 4 | 汽车 | 加油、停车、保养、洗车、过路费等 |
| 5 | 日用百货 | 超市购物、日用品采购等 |
| 6 | 备婚 | 婚庆服务、婚纱摄影、婚礼用品等 |
| 7 | 人情 | 红包、礼金、请客吃饭等 |
| 8 | 交通 | 打车、公交、地铁、火车票等 |
| 9 | 数码电器 | 手机、电脑、家电、数码配件等 |
| 10 | 文化休闲 | 电影、演出、游戏、书籍等 |
| 11 | 服饰 | 服装、鞋帽、配饰等 |
| 12 | 通讯 | 话费、宽带、流量包等 |
| 13 | 家居家装 | 家具、装修材料、家居用品等 |
| 14 | 水电燃气费 | 水费、电费、燃气费等 |
| 15 | 医疗 | 药品、挂号费、体检等 |
| 16 | 美妆护肤 | 化妆品、护肤品、美容服务等 |
| 17 | 运动户外 | 健身房、运动装备、户外用品等 |
| 18 | 学习 | 书籍、课程、培训等 |
| 19 | 其他支出 | 无法归类的其他支出 |

### 2.3 数据处理规则

- **收入/支出区分**：仅统计收入和支出两类交易
- **分类匹配**：通过关键词匹配和机器学习相结合的方式
- **异常处理**：无法匹配的交易归入"其他支出"类别

### 3.1 技术选型

| 层面 | 技术选择 | 说明 |
|-----|---------|------|
| 编程语言 | Python 3.10+ | 丰富的数据处理库生态 |

### 3.3 核心模块设计

#### 3.3.1 文件解析模块
- 支持解析支付宝账单 Excel 格式
- 支持解析微信账单 Excel/CSV 格式
- 支持解析主流银行账单格式
- 自动识别文件来源并选择对应解析器

#### 3.3.2 分类引擎模块
- **关键词匹配**：基于预设关键词库进行初步分类
- **规则引擎**：支持正则表达式匹配
- **机器学习（可选）**：训练分类模型提高准确率

#### 3.3.3 数据导出模块
- 输出格式：Excel (.xlsx)
- 包含分类汇总表和明细表
- 支持按月/按年统计

## 五、数据流程

```
1. 用户导入交易流水文件
        ↓
2. 系统自动识别文件类型（支付宝/微信/银行）
        ↓
3. 调用对应解析器提取交易数据
        ↓
4. 分类引擎对每条交易进行分类
        ↓
5. 用户可手动调整分类结果
        ↓
6. 导出整理后的 Excel 文件
```

## 六、输入输出规范

### 6.1 支持的输入格式

| 来源 | 格式 | 关键字段 |
|-----|------|---------|
| 支付宝 | .xlsx | 交易时间、交易分类、交易对方、商品说明、金额、收/支 |
| 微信 | .xlsx/.csv | 交易时间、交易类型、交易对方、商品、金额、收/支 |
| 银行 | .xlsx/.csv | 交易日期、摘要、对方户名、金额、借/贷 |

## 七、开发计划

### 7.1 里程碑

| 阶段 | 内容 | 预计周期 |
|-----|------|---------|
| 第一阶段 | 基础框架搭建、文件解析功能 | 1 周 |
| 第二阶段 | 分类引擎开发、关键词库建设 | 1 周 |
| 第三阶段 | GUI 界面开发 | 1 周 |
| 第四阶段 | 测试优化、打包发布 | 1 周 |

### 7.2 技术风险

| 风险 | 影响 | 应对措施 |
|-----|-----|---------|
| 不同平台账单格式差异大 | 解析复杂度高 | 采用适配器模式，逐个平台适配 |
| 分类准确率不足 | 用户体验差 | 持续优化关键词库，支持用户反馈 |
| 文件编码问题 | 读取失败 | 自动检测编码格式 |

## 九、总结
